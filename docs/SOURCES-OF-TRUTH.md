# Sources of Truth

The 10-pillar plan's invariant I1: **every piece of state has exactly one source of truth (SoT); everything else is a denormalized view**. Reconcile heals views from the SoT, never the other way around. This document is the registry.

When you add a new piece of persistent state, you **must** add a row here before the code merges. When you change a SoT, write an ADR.

## Adding a new row

1. Identify the SoT. If you can't name a single location, the design has a bug ŌĆö fix the design first.
2. List every place the same data appears as a denormalized view.
3. Specify the heal direction (which way reconcile flows) and which pass owns it.
4. If the new state has no existing reconcile coverage, add the pass to the plan in the same PR.
5. Update the registry in alphabetical order within section if it gets long; for now chronological-by-addition is fine.

## Anti-patterns

- **"Both are SoT" entries.** If the vault and the ledger can both authoritatively answer the same question, one of them is wrong. Pick.
- **Heal-in-both-directions.** Reconcile is a one-way operation per row. Bidirectional heal is a polite name for data corruption.
- **Implicit SoTs.** A piece of state that no row in this table covers is not "obviously the database" ŌĆö it's undocumented. Write the row.
